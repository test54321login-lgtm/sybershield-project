WHAT HAPPENS WHEN USERS LOG IN/SIGN UP
═════════════════════════════════════════════════════════════════════════════

1. User Submits Form
   └─ Frontend validates input (email format, password length, etc.)

2. API Validates Again
   └─ Server-side validation prevents malicious requests

3. Password Hashing
   └─ Plain password → Bcryptjs hash (12 salt rounds)
   └─ Hash stored in MongoDB (never the original password)

4. JWT Token Generated
   └─ Contains: user ID, email, username, expiration time
   └─ Signed with JWT_SECRET
   └─ Valid for 7 days

5. Token Sent to Frontend
   └─ Stored in browser localStorage
   └─ Sent with every API request

6. Server Verifies Token
   └─ Checks signature using JWT_SECRET
   └─ Checks expiration time
   └─ If valid: request proceeds
   └─ If invalid/expired: user must log in again

UNDERSTANDING THE JWT TOKEN
═════════════════════════════════════════════════════════════════════════════

A JWT token looks like this (three parts separated by dots):

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.
eyJpZCI6IjUwN2YxZjc3YmNmODZjZDc5OTQzOTAxMSIsImVtYWlsIjoiY3liZXJAZXhhbXBs
ZS5jb20iLCJ1c2VybmFtZSI6ImN5YmVydXNlciIsImlhdCI6MTcxMjYwNjQwMCwiZXhwIjoxNzEzMjExMjAwfQ.
xY9zB4qLpMnOp_lRsT7vG6hJkLmNqRsUvWxYzAbCdEf

PART 1 - Header:
  {
    "alg": "HS256",
    "typ": "JWT"
  }

PART 2 - Payload (User Data):
  {
    "id": "507f1f77bcf86cd799439011",
    "email": "cyber@example.com",
    "username": "cyberuser",
    "iat": 1712606400,      // Issued at (time token was created)
    "exp": 1713211200       // Expiration (7 days later)
  }

PART 3 - Signature:
  HMACSHA256(
    base64UrlEncode(header) + "." + base64UrlEncode(payload),
    JWT_SECRET
  )
  
The signature proves the token hasn't been tampered with!

HOW THE JWT_SECRET PROTECTS YOUR USERS
═════════════════════════════════════════════════════════════════════════════

When a token is created:
  1. User data is encoded (ID, email, username, timestamps)
  2. Header is encoded (algorithm type)
  3. Both are hashed using JWT_SECRET
  4. Signature is created and appended
  5. Final token is sent to user

When a token is verified:
  1. Server receives token from user
  2. Server re-hashes the header + payload using JWT_SECRET
  3. Server compares calculated hash with token's signature
  4. If they match → token is valid and hasn't been modified
  5. If they don't match → token is invalid (tampered with)

If someone tries to fake a token:
  ✗ They can modify the payload (add admin role, etc.)
  ✗ BUT they can't create a valid signature without JWT_SECRET
  ✗ Server will detect the mismatch and reject the token
  ✗ Attacker would need to know JWT_SECRET (which they don't!)
